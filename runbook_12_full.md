# Recover without get-wal

## Objective

Recover the latest base backup of the primary *not* using `get-wal` as recovery options.


## Requirements:

- rb10


## Make sure Postgres is down and PGDATA is empty on the restore server

```bash
# Execute within the restore container
docker exec --user postgres -i pgrestore bash << EOF
# Stop the PostgreSQL 17 service
/usr/pgsql-17/bin/pg_ctl -D /var/lib/pgsql/17/data stop -m fast

# Clean the PGDATA directory
rm -rf /var/lib/pgsql/17/data/*
EOF
```


## Run barman recover

We have configured `recovery_options = 'get-wal'`:

```
barman@9c99447bf70f:~$ barman show-server rb08 | grep recovery_options
	recovery_options: RecoveryOptions({'get-wal'})
```

But we can override this setting by adding the `--no-get-wal` argument to `barman recover`:

```bash
barman recover --no-get-wal \
  --remote-ssh-command "ssh postgres@pgrestore" \
  rb08 latest /var/lib/pgsql/17/data/
```

Output will be similar to:

```
[barman@1c37152b8827 ~]$ barman recover --no-get-wal \
  --remote-ssh-command "ssh postgres@pgrestore" \
  rb08 latest /var/lib/pgsql/17/data/
Starting remote restore for server rb08 using backup 20260311T174245
Destination directory: /var/lib/pgsql/17/data/
Remote command: ssh postgres@pgrestore
Using safe horizon time for smart rsync copy: 2026-03-11 17:42:45.418198+00:00
Copying the base backup.
Copying required WAL segments.
Generating archive status files
Identify dangerous settings in destination directory.

IMPORTANT
These settings have been modified to prevent data losses

postgresql.auto.conf line 3: archive_command = false

Restore operation completed (start time: 2026-03-11 20:20:29.041341+00:00, elapsed time: 43 seconds)
```

Note the `Copying required WAL segments` step. It means WAL files are copied together to the target PGDATA inside the `pg_wal` directory, so they won't need to be fetched using `restore_command`.

Barman has also created a file called `recovery.signal` and has filled recovery settings in `postgresql.auto.conf`:

```
postgres@5333f7e45b0f:~/13/main$ cat postgresql.auto.conf
# Do not edit this file manually!

archive_mode = 'on'
#BARMAN#archive_command = 'barman-wal-archive pgbkp1 rb08 %p'
archive_command = false
```

If it was PostgreSQL 11 or older, the recovery settings would have been written to the file `recovery.conf` instead.
